# 人物角色构建器

引导用户完成小说人物的创建：检索现有项目 → 确认目标项目 → 读取世界观上下文 → 专家讨论配置 → 填充人物信息 → 生成人物档案文件。

## 工作流程

```
用户请求创建人物
       ↓
┌──────────────────┐
│ 1. 检索现有项目   │  ← 扫描工作目录下的小说项目
└──────────────────┘
       ↓
┌──────────────────┐
│ 2. 确认目标项目   │  ← 询问用户选择或指定项目
└──────────────────┘
       ↓
┌──────────────────┐
│ 3. 读取世界观     │  ← 读取项目世界观文件作为上下文
└──────────────────┘
       ↓
┌──────────────────┐
│ 4. 专家讨论配置   │  ← 调用 project-discussion skill 讨论人物配置
└──────────────────┘
       ↓
┌──────────────────┐
│ 5. 确认角色类型   │  ← 主角/反派/配角/龙套
└──────────────────┘
       ↓
┌──────────────────┐
│ 6. 填充人物信息   │  ← 逐项填写人物模板
└──────────────────┘
       ↓
┌──────────────────┐
│ 7. 生成人物档案   │  ← 创建人物档案文件（含YAML元数据）
└──────────────────┘
       ↓
   人物就绪
```

## 第一步：检索现有项目

### 项目识别标准

扫描工作目录，识别符合以下条件的小说项目：

1. **目录结构特征**：
   - 包含 `项目信息/项目配置.md` 文件
   - 或包含 `设定/人物档案/` 目录
   - 或包含 `正文/` 目录

2. **识别方法**：
   - 使用 Glob 工具搜索 `**/项目配置.md`
   - 使用 LS 工具列出工作目录下的文件夹
   - 检查每个文件夹是否包含小说项目特征

### 项目信息提取

对于每个识别到的项目，提取以下信息：
- 项目名称（目录名）
- 小说类型（从项目配置中读取）
- 当前人物数量（统计人物档案目录）

## 第二步：确认目标项目

### 询问策略

使用 AskUserQuestion 工具询问用户：

1. **如果找到现有项目**：
   - 列出所有找到的项目供用户选择
   - 显示每个项目已有的人物数量
   - 提供"新建项目"选项

2. **如果未找到项目**：
   - 询问用户是否要新建项目
   - 或指定已有项目的路径

### 选项格式示例

```
问题：请选择要添加人物的项目

选项：
- 苍穹之巅（玄幻）- 已有3个人物
- 星辰大海（科幻）- 无人物
- 新建项目
```

### 用户指定项目

用户可以通过以下方式指定项目：
- 直接说出项目名称
- 提供项目路径
- 描述项目特征（如"那个玄幻小说"）

## 第三步：读取世界观上下文

**此步骤确保人物设定与世界观保持一致！**

### 读取目标

确认目标项目后，读取该项目的世界观文件作为人物创建的上下文：

1. **世界观文件位置**：
   - `{项目目录}/设定/世界观/` 目录下的所有 `.md` 文件
   - 优先读取 `世界观-主世界.md`

2. **读取方法**：
   - 使用 Glob 工具搜索 `{项目目录}/设定/世界观/*.md`
   - 使用 Read 工具读取找到的世界观文件

### 上下文应用

将世界观信息作为人物创建的重要参考：

| 世界观要素 | 人物设定关联 |
|------------|--------------|
| 时代背景 | 人物的背景故事、身份职业 |
| 地点设定 | 人物的出身地、活动范围 |
| 规则体系 | 人物的能力/技能设计 |
| 社会结构 | 人物的阵营/立场、身份地位 |
| 科技/超自然体系 | 人物的能力来源、等级划分 |
| 文化风俗 | 人物的说话风格、习俗禁忌 |

### 无世界观处理

如果项目暂无世界观文件：
- 提示用户可以先创建世界观
- 或继续创建人物，后续再关联世界观

## 第四步：专家讨论配置

**在获取项目上下文后执行此步骤！**

调用 `project-discussion` skill，聘请世界级专家对人物创建进行深度讨论。

### 讨论上下文

讨论时需要结合以下上下文信息：
- 项目配置（小说类型、主题、目标读者等）
- 世界观设定（时代背景、规则体系、社会结构等）
- 人物模板中的所有要素

### 讨论议题

讨论内容应覆盖人物模板中的所有要素：

| 讨论维度 | 配置项 | 讨论要点 |
|----------|--------|----------|
| 身份定位 | 姓名、别名/称号、身份/职业、阵营/立场、角色类型 | 命名艺术、身份象征意义、阵营选择策略 |
| 外在形象 | 外貌特征 | 视觉记忆点、外貌与性格的呼应、标志性特征设计 |
| 内在特质 | 性格特点 | 性格层次、性格矛盾、性格成长弧线 |
| 能力设计 | 能力/技能 | 能力与剧情的关联、能力平衡、成长空间 |
| 故事深度 | 背景故事、目标/动机 | 人物弧光、动机合理性、过去与现在的连接 |
| 人物网络 | 人物关系 | 关系网设计、关系冲突、关系演变 |
| 语言特色 | 口头禅/说话风格 | 语言辨识度、风格一致性、语言与性格匹配 |
| 人物缺陷 | 弱点/恐惧 | 弱点设计、恐惧来源、缺陷与成长的关联 |

### 执行方式

```
调用 project-discussion skill
     ↓
根据项目类型和世界观动态选择专家
     ↓
展示拟聘请专家名单
     ↓
用户确认专家
     ↓
执行多轮讨论
     ↓
输出讨论纪要
     ↓
基于讨论结果进入下一步
```

### 专家选择原则

**根据具体项目动态决定专家人选，不预设固定专家角色！**

专家选择应考虑：
- **小说类型**：玄幻/科幻/都市/历史等不同类型需要不同领域专家
- **世界观特点**：修炼体系、科技设定、历史背景等
- **目标读者**：不同读者群体的人物偏好
- **角色类型**：主角/反派/配角的设计重点不同

**注意**：必须等待用户确认专家名单后才能开始讨论！

## 第五步：确认角色类型

### 角色类型分类

| 类型 | 说明 | 创建优先级 |
|------|------|------------|
| 主角 | 故事核心人物，视角担当 | 最高 |
| 反派 | 与主角对立的主要对手 | 高 |
| 配角 | 重要辅助角色，推动剧情 | 中 |
| 龙套 | 背景角色，功能性存在 | 低 |

### 询问策略

询问用户要创建的角色类型，并根据类型调整后续问题：

- **主角**：需要详细填写所有字段，特别关注背景故事、目标动机、人物弧光
- **反派**：重点填写与主角的关系、动机、弱点
- **配角**：重点填写与主角的关系、功能定位
- **龙套**：只需填写基本信息和功能定位

## 第六步：填充人物信息

读取 `templates/人物模板.md`，逐项引导用户填写：

### 基础信息（必填）

| 字段 | 说明 | 示例 |
|------|------|------|
| 姓名 | 角色名称 | 林霄 |
| 别名/称号 | 称呼方式 | 苍穹剑仙、废材师兄 |
| 身份/职业 | 社会身份 | 苍穹宗弟子 |
| 阵营/立场 | 正邪归属 | 正道/中立/魔道 |
| 角色类型 | 角色定位 | 主角/反派/配角/龙套 |

### 外貌与性格（必填）

| 字段 | 说明 |
|------|------|
| 外貌特征 | 容貌、身材、穿着、标志性特征 |
| 性格特点 | 性格关键词、行为习惯 |

### 能力与背景（建议填写）

| 字段 | 说明 |
|------|------|
| 能力/技能 | 特殊能力、战斗技能、专业技能 |
| 背景故事 | 身世来历、重要经历 |

### 关系与特征（可选）

| 字段 | 说明 |
|------|------|
| 人物关系 | 与其他角色的关系 |
| 口头禅/说话风格 | 语言特点 |
| 目标/动机 | 行为驱动力 |
| 弱点/恐惧 | 角色缺陷 |

**执行方式**：使用 AskUserQuestion 工具逐项收集信息，或让用户直接提供完整设定。

## 第七步：生成人物档案

### 目录创建

**在写入文件前，必须确保目录存在：**

```
目标目录：{项目目录}/设定/人物档案/

创建逻辑：
1. 检查 {项目目录}/设定/ 是否存在
2. 如不存在，创建 设定/ 目录
3. 检查 {项目目录}/设定/人物档案/ 是否存在
4. 如不存在，创建 人物档案/ 目录
5. 写入文件
```

### 文件创建规则

1. **存储位置**：
   - 存储到 `{项目目录}/设定/人物档案/`

2. **文件命名**：
   - 单人物文件：`人物-{姓名}.md`
   - 批量人物文件：`人物档案-{类型}.md`（如：人物档案-主角.md）

3. **文件内容**：
   - 基于模板填充用户设定
   - 包含 YAML 元数据（见下方规范）
   - 多人物时用 `---` 分隔
   - 保持结构清晰、层次分明

## YAML 元数据规范

### 元数据结构

```yaml
---
# === 核心字段（总是读取）===
type: character
name: {姓名}
role: {protagonist/antagonist/supporting/minor}

# === 角色驱动字段（大纲生成必读）===
desire: {核心欲望}
weakness: {心理弱点}
arc: {成长弧线}

# === 索引字段（过滤时读取）===
volumes: [{涉及的卷}]
first_appear: {首次出场章节}

# === 关联字段（关联时读取）===
relationships:
  {角色名}: {关系}

# === 摘要字段（快速了解）===
summary: {一句话摘要}
---
```

### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| type | string | 是 | 固定值 `character` |
| name | string | 是 | 角色姓名 |
| role | string | 是 | 角色类型：protagonist/antagonist/supporting/minor |
| desire | string | 是 | 核心欲望，驱动故事的关键 |
| weakness | string | 是 | 心理弱点，成长弧线的关键 |
| arc | string | 是 | 成长弧线：起点→终点 |
| volumes | array | 否 | 涉及的卷号列表 |
| first_appear | number | 否 | 首次出场章节 |
| relationships | object | 否 | 人物关系映射 |
| summary | string | 是 | 一句话摘要，用于其他 Skill 快速了解 |

### 元数据用途

| 字段 | 被谁读取 | 读取目的 |
|------|----------|----------|
| type | 所有 Skill | 确认文件类型 |
| name | 所有 Skill | 标识角色 |
| role | 大纲生成器 | 筛选主角/反派/配角 |
| desire | 大纲生成器、场景生成器 | 角色驱动故事设计 |
| weakness | 大纲生成器、场景生成器 | 设计冲突和成长 |
| arc | 大纲生成器 | 规划角色成长轨迹 |
| volumes | 大纲生成器 | 筛选当前卷相关角色 |
| first_appear | 大纲生成器 | 规划出场时机 |
| relationships | 场景生成器 | 设计人物互动 |
| summary | 所有 Skill | 快速了解内容，决定是否深度读取 |

## 渐进式披露读取策略

其他 Skill 读取人物文件时，应遵循以下策略：

### 阶段1：扫描（~50字符）

```
读取：type, name, role
目的：确认是人物文件，了解角色类型
决策：是否需要继续读取
```

### 阶段2：角色驱动（~100字符）

```
读取：desire, weakness, arc
目的：获取角色驱动的核心信息
决策：如何让角色驱动故事
```

### 阶段3：关联（~150字符）

```
读取：volumes, first_appear, relationships
目的：建立人物关联
决策：需要关联哪些其他文件
```

### 阶段4：深度（全文）

```
读取：Markdown 正文
目的：获取完整人物设定
决策：如何使用这些信息
```

## 完成确认

人物创建完成后，向用户展示：

1. **项目信息**：关联的项目名称
2. **人物概览**：姓名、类型、核心特征
3. **文件位置**：生成的文件路径
4. **下一步建议**：
   - 继续添加其他人物
   - 完善人物关系图
   - 规划人物故事线

## 注意事项

- 人物设定要与世界观保持一致
- 主角和反派需要有明确的目标和动机
- 人物性格要有矛盾和成长空间
- 人物关系要清晰，避免混乱
- 如果项目已有人物，建议先展示现有人物列表
- **必须确保目录存在后再写入文件**
