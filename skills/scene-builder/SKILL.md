# 场景构建器

> "场景是故事的最小单位，每个场景都必须有价值变化——从正面到负面，或从负面到正面。没有变化的场景就是废话。"
> — 罗伯特·麦基《故事》

引导用户完成小说场景的创建：检索现有项目 → 确认目标项目 → 读取项目上下文 → 专家讨论配置 → 设计场景结构 → 生成场景文件。

## 工作流程

```
用户请求创建场景
       ↓
┌──────────────────┐
│ 1. 检索现有项目   │  ← 扫描工作目录下的小说项目
└──────────────────┘
       ↓
┌──────────────────┐
│ 2. 确认目标项目   │  ← 询问用户选择或指定项目
└──────────────────┘
       ↓
┌──────────────────┐
│ 3. 读取项目上下文 │  ← 读取世界观、人物档案作为上下文
└──────────────────┘
       ↓
┌──────────────────┐
│ 4. 专家讨论配置   │  ← 调用 project-discussion skill 讨论场景配置
└──────────────────┘
       ↓
┌──────────────────┐
│ 5. 确认场景类型   │  ← 对话/动作/过渡/高潮
└──────────────────┘
       ↓
┌──────────────────┐
│ 6. 设计场景结构   │  ← 价值变化、冲突、节拍
└──────────────────┘
       ↓
┌──────────────────┐
│ 7. 生成场景文件   │  ← 创建场景设计文档（含YAML元数据）
└──────────────────┘
       ↓
   场景就绪
```

## 第一步：检索现有项目

### 项目识别标准

扫描工作目录，识别符合以下条件的小说项目：

1. **目录结构特征**：
   - 包含 `项目信息/项目配置.md` 文件
   - 或包含 `大纲/` 目录
   - 或包含 `正文/` 目录

2. **识别方法**：
   - 使用 Glob 工具搜索 `**/项目配置.md`
   - 使用 LS 工具列出工作目录下的文件夹

### 项目信息提取

对于每个识别到的项目，提取以下信息：
- 项目名称（目录名）
- 小说类型（从项目配置中读取）
- 当前章节/场景情况

## 第二步：确认目标项目

### 询问策略

使用 AskUserQuestion 工具询问用户：

1. **如果找到现有项目**：
   - 列出所有找到的项目供用户选择
   - 显示每个项目的基本信息

2. **如果未找到项目**：
   - 询问用户是否要新建项目
   - 或指定已有项目的路径

### 选项格式示例

```
问题：请选择要创建场景的项目

选项：
- 苍穹之巅（玄幻）- 第一卷进行中
- 星辰大海（科幻）- 大纲阶段
- 新建项目
```

## 第三步：读取项目上下文

**此步骤确保场景设定与项目整体保持一致！**

### 读取目标

确认目标项目后，读取该项目的相关文件作为场景创建的上下文：

1. **世界观文件**：
   - 位置：`{项目目录}/设定/世界观/` 目录下的所有 `.md` 文件
   - 用途：确保场景的环境、氛围、规则与世界观一致

2. **人物档案**：
   - 位置：`{项目目录}/设定/人物档案/` 目录下的所有 `.md` 文件
   - 用途：确保场景中人物的行为、对话、关系与人物设定一致

3. **项目配置**：
   - 位置：`{项目目录}/项目信息/项目配置.md`
   - 用途：了解小说类型、写作风格等基础信息

### 读取方法

- 使用 Glob 工具搜索相关目录下的 `.md` 文件
- 使用 Read 工具读取找到的文件

### 上下文应用

| 上下文类型 | 场景设计关联 |
|------------|--------------|
| 世界观-时代背景 | 场景的时间设定 |
| 世界观-地点设定 | 场景的地点选择 |
| 世界观-规则体系 | 场景中人物能力的使用 |
| 人物-性格特点 | 场景中人物的行为反应 |
| 人物-人物关系 | 场景中人物互动的方式 |
| 人物-说话风格 | 场景对话的语言风格 |
| 项目配置-写作风格 | 场景的描写风格 |

### 无上下文处理

如果项目暂无相关文件：
- 提示用户可以先创建世界观或人物档案
- 或继续创建场景，后续再关联

## 第四步：专家讨论配置

**在获取项目上下文后执行此步骤！**

调用 `project-discussion` skill，聘请世界级专家对场景创建进行深度讨论。

### 讨论上下文

讨论时需要结合以下上下文信息：
- 项目配置（小说类型、主题、写作风格等）
- 世界观设定（时代背景、地点、规则体系等）
- 人物档案（相关人物的性格、关系、说话风格等）
- 场景模板中的所有要素

### 讨论议题

讨论内容应覆盖场景模板中的所有要素：

| 讨论维度 | 配置项 | 讨论要点 |
|----------|--------|----------|
| 基础定位 | 场景名称、所属章节、场景类型、POV视角 | 场景命名、类型选择策略、视角选择的影响 |
| 价值变化 | 初始价值、最终价值、转折点 | 价值极性设计、翻转时机、变化幅度 |
| 冲突设计 | 场景目标、阻碍因素、场景风险、场景结果 | 目标明确性、阻碍层次、风险真实感、结果影响 |
| 环境氛围 | 时间、地点、环境描写、氛围基调 | 环境与情绪的呼应、氛围强化冲突 |
| 人物对话 | 出场人物、核心对话、潜台词 | 人物功能、对话张力、潜台词设计 |
| 节拍结构 | 节拍分解 | 节奏控制、情绪曲线、信息分布 |
| 写作策略 | 必须呈现、可以省略、伏笔/呼应 | 信息取舍、伏笔技巧、前后呼应 |

### 执行方式

```
调用 project-discussion skill
     ↓
根据项目类型和上下文动态选择专家
     ↓
展示拟聘请专家名单
     ↓
用户确认专家
     ↓
执行多轮讨论
     ↓
输出讨论纪要
     ↓
基于讨论结果进入下一步
```

### 专家选择原则

**根据具体项目动态决定专家人选，不预设固定专家角色！**

专家选择应考虑：
- **小说类型**：玄幻/科幻/都市/历史等不同类型需要不同领域专家
- **场景类型**：对话场景/动作场景/高潮场景的设计重点不同
- **世界观特点**：修炼体系、科技设定、历史背景等
- **目标读者**：不同读者群体的场景偏好

**注意**：必须等待用户确认专家名单后才能开始讨论！

## 第五步：确认场景类型

### 场景类型分类（基于麦基理论）

| 类型 | 说明 | 核心要素 |
|------|------|----------|
| 对话场景 | 以对话推动的场景 | 信息交换、立场冲突、关系变化 |
| 动作场景 | 以行动推动的场景 | 目标、障碍、风险、结果 |
| 过渡场景 | 连接主要场景 | 时间流逝、空间转换、情绪铺垫 |
| 高潮场景 | 情感/冲突的顶点 | 最大风险、最终抉择、价值翻转 |

### 场景功能分类

| 功能 | 说明 |
|------|------|
| 推进主线 | 推动主要情节发展 |
| 揭示人物 | 展现角色性格、动机、秘密 |
| 建立关系 | 发展角色之间的关系 |
| 传递信息 | 向读者传递必要信息 |
| 制造悬念 | 埋设伏笔、制造期待 |

## 第六步：设计场景结构

### 核心理论：价值变化

**麦基原则**：每个场景都必须有价值变化。

```
场景开始时的价值状态
        ↓
    冲突发生
        ↓
场景结束时的价值状态（必须与开始时不同）
```

**价值极性**：
- 正面 → 负面（希望 → 绝望）
- 负面 → 正面（危险 → 安全）
- 正面 → 更正面（成功 → 更大成功）
- 负面 → 更负面（失败 → 更大失败）

### 场景节拍设计

每个场景由多个"节拍"组成：

| 节拍 | 说明 |
|------|------|
| 开场 | 确立场景的初始价值状态 |
| 发展 | 冲突逐步升级 |
| 转折 | 价值发生翻转的关键时刻 |
| 收尾 | 确立新的价值状态，为下一场景铺垫 |

### 读取模板

读取 `templates/场景模板.md`，逐项引导用户填写：

#### 基础信息（必填）

| 字段 | 说明 | 示例 |
|------|------|------|
| 场景名称 | 场景的标识 | 林霄被逐出宗门 |
| 所属章节 | 场景所在位置 | 第一卷第三章 |
| 场景类型 | 场景的性质 | 高潮场景 |
| POV视角 | 叙事视角 | 林霄 |

#### 价值变化（必填）

| 字段 | 说明 |
|------|------|
| 初始价值 | 场景开始时的状态 |
| 最终价值 | 场景结束时的状态 |
| 转折点 | 价值翻转的关键时刻 |

#### 冲突设计（必填）

| 字段 | 说明 |
|------|------|
| 场景目标 | 角色想要什么 |
| 阻碍因素 | 什么阻止角色达成目标 |
| 场景风险 | 如果失败会发生什么 |
| 场景结果 | 最终结果是什么 |

#### 环境与氛围（建议填写）

| 字段 | 说明 |
|------|------|
| 时间 | 场景发生的时间 |
| 地点 | 场景发生的地点 |
| 环境 | 天气、光线、温度等 |
| 氛围 | 整体情绪基调 |

#### 人物与对话（建议填写）

| 字段 | 说明 |
|------|------|
| 出场人物 | 场景中的角色 |
| 核心对话 | 关键对话要点 |
| 潜台词 | 对话背后的真实含义 |

**执行方式**：使用 AskUserQuestion 工具逐项收集信息，或让用户直接提供完整设定。

## 第七步：生成场景文件

### 目录创建

**在写入文件前，必须确保目录存在：**

```
目标目录：{项目目录}/大纲/场景设计/

创建逻辑：
1. 检查 {项目目录}/大纲/ 是否存在
2. 如不存在，创建 大纲/ 目录
3. 检查 {项目目录}/大纲/场景设计/ 是否存在
4. 如不存在，创建 场景设计/ 目录
5. 写入文件
```

### 文件创建规则

1. **存储位置**：
   - 存储到 `{项目目录}/大纲/场景设计/`
   - 或存储到 `{项目目录}/正文/{卷名}/` 作为章节参考

2. **文件命名**：
   - 单场景文件：`场景-{场景名称}.md`
   - 章节场景：`第X章-场景设计.md`

3. **文件内容**：
   - 基于模板填充用户设定
   - 包含 YAML 元数据（见下方规范）
   - 保持结构清晰、层次分明

## YAML 元数据规范

### 元数据结构

```yaml
---
# === 核心字段（总是读取）===
type: scene
name: {场景名称}
chapter: {章节号}
volume: {卷号}

# === 价值变化字段（场景核心）===
value_start: {正面/负面}
value_end: {正面/负面}
value_turn: {转折点}

# === 关联字段（关联时读取）===
characters: [{出场人物}]
location: {地点}

# === 摘要字段（快速了解）===
summary: {一句话摘要}
---
```

### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| type | string | 是 | 固定值 `scene` |
| name | string | 是 | 场景名称 |
| chapter | number | 是 | 章节号 |
| volume | number | 是 | 卷号 |
| value_start | string | 是 | 初始价值状态：正面/负面 |
| value_end | string | 是 | 最终价值状态：正面/负面 |
| value_turn | string | 是 | 价值转折点描述 |
| characters | array | 否 | 出场人物列表 |
| location | string | 否 | 场景地点 |
| summary | string | 是 | 一句话摘要，用于其他 Skill 快速了解 |

### 元数据用途

| 字段 | 被谁读取 | 读取目的 |
|------|----------|----------|
| type | 所有 Skill | 确认文件类型 |
| name | 所有 Skill | 标识场景 |
| chapter | 大纲生成器、正文写作 | 定位场景位置 |
| volume | 大纲生成器 | 筛选当前卷相关场景 |
| value_start | 大纲生成器、正文写作 | 了解场景价值起点 |
| value_end | 大纲生成器、正文写作 | 了解场景价值终点 |
| value_turn | 大纲生成器、正文写作 | 设计场景转折 |
| characters | 场景生成器 | 关联人物设定 |
| location | 场景生成器 | 关联世界观地点 |
| summary | 所有 Skill | 快速了解内容，决定是否深度读取 |

## 渐进式披露读取策略

其他 Skill 读取场景文件时，应遵循以下策略：

### 阶段1：扫描（~50字符）

```
读取：type, name, chapter, volume
目的：确认是场景文件，了解位置
决策：是否需要继续读取
```

### 阶段2：价值核心（~100字符）

```
读取：value_start, value_end, value_turn
目的：获取场景的价值变化核心
决策：如何设计场景转折
```

### 阶段3：关联（~150字符）

```
读取：characters, location
目的：建立场景关联
决策：需要关联哪些其他文件
```

### 阶段4：深度（全文）

```
读取：Markdown 正文
目的：获取完整场景设计
决策：如何使用这些信息
```

## 完成确认

场景创建完成后，向用户展示：

1. **项目信息**：关联的项目名称
2. **场景概览**：名称、类型、价值变化
3. **文件位置**：生成的文件路径
4. **下一步建议**：
   - 继续设计下一场景
   - 检查场景间的衔接
   - 开始撰写正文

## 注意事项

### 麦基的警告

> "如果一个场景没有改变价值，删掉它。"

- 每个场景必须有明确的价值变化
- 没有冲突的场景是死场景
- 场景风险必须真实可信

### 特鲁比的建议

- 场景是角色内在冲突的外化
- 场景空间应该强化冲突
- 好的场景有多层含义

### 实践要点

- 场景设定要与世界观保持一致
- 场景目标要与角色动机一致
- 场景结果要推动故事发展
- 注意场景间的节奏变化
- **必须确保目录存在后再写入文件**
